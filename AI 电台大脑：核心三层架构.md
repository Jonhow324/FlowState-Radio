目前你直接把长期偏好写死在 LLM 的 System Prompt（MD文档）里，会导致两个痛点：一是 **LLM 无法实时检索我的庞大的 歌单**；二是 **LLM 对“根据场景选歌”的泛化能力不稳定**。

要打造一个聪明、响应迅速的“电台大脑”，建议采用 **“协同过滤/向量检索 + LLM 场景理解与润色”** 的双层架构。以下是为你设计的进化方案：

## 🧠 AI 电台大脑：核心三层架构

不要让 LLM 孤军奋战，把它定位为“节目制作人兼主持人”，而不是“音乐搜索引擎”。

```
[输入层: 天气/时间/日程] ---> [大脑第1层: LLM 场景向量生成]
                                  |
                                  v
[听歌习惯/全量歌单] ---------> [大脑第2层: 向量数据库/候选集检索] ---> [初步歌单 (10-20首)]
                                  |
                                  v
                            [大脑第3层: LLM 最终筛选 & 串词生成] ---> [输出: 播放列表 + 语音TTS]
```

### 1. 数据层：向量化你的听歌习惯（建立“音乐记忆”）

不能把整张歌单塞给 LLM，你需要把你的歌单和听歌习惯**数字化（Embedding）**。

- **歌单向量化：** 把你歌单里的每首歌（歌名、歌手、风格标签、歌词大意，甚至你对它的评分/听歌频率）组合成一段文本，通过 Embedding 模型（如 `text-embedding-3-small`）转换成向量，存入本地向量数据库（如 ChromaDB 或 Qdrant）。
- **用户画像分时段固化：** 不要在 MD 里写“我喜欢周杰伦”，而是写成场景关联。例如：
  - `{"场景": "工作/深夜", "偏好": "Lo-Fi, 纯音乐, 乐器, 告五人", "避雷": "重金属"}`

### 2. 检索层：基于 Context 的“粗筛”（RAG 模式）

当你的 App 触发定时刷新或用户打开 App 时，大脑开始运转：

1. **解析当前上下文：** 获取当前数据，例如：`星期一 | 早上 08:30 | 下雨 🌧️ | 日程：9:30 有一个紧急交付会议`。

2. **生成检索 Prompt：** 让一个轻量、便宜的 LLM（或直接用模板）将这些要素转化为一个“音乐意图描述”：

   > *"一个下雨天的周一早晨，用户正准备迎接一场压力较大的紧急会议，需要一些能舒缓焦虑、提神且不过分嘈杂的背景音乐。"*

3. **向量检索：** 将这段描述同样转换为向量，去你的向量数据库里计算相似度，召回 **30-50首** 匹配你听歌习惯的歌曲。**（这就解决了没有接入你歌单的问题）**。

### 3. 生成层：LLM 的“精选”与“电台串词”（核心灵魂）

现在，你手里有一组**精准踩在你审美上、且符合当前大方向**的候选歌单了。接下来把接力棒交给能力最强的 LLM

给 LLM 的输入包含三部分：

1. **当前严苛的上下文**（时间、天气、具体日程）。
2. **上一步粗筛出来的 20 首备选歌曲清单**（带风格标签）。
3. **电台DJ的角色设定**。

#### 📝 推荐给你的 Prompt 架构示例：

Markdown

```
# Role
你是一个高情商、声音温柔、充满洞察力的 AI 电台 DJ。

# Context
- 时间: 2026-06-08 08:30 (Monday)
- 天气: 暴雨，气温 18°C
- 今日紧要日程: 09:30 团队交付评审会 (压力较大)

# Candidates (经过你歌单粗筛后的候选歌曲)
1. 《Raincoats》 - Ocean Hope (Dream Pop, 舒缓)
2. 《Balm》 - Brockhew (Ambient, 专注)
3. 《晴天》 - 周杰伦 (流行, 怀旧)  <-- 比如这首可能就不适合现在的暴雨和压力，LLM 负责剔除
... (放10-15首)

# Task
1. 从 Candidates 中精准挑选出 3-5 首最适合当前场景和用户心态的歌曲顺序。
2. 为选出的每首歌生成一段 50 字以内的“电台主持串词”。
3. 串词要求：必须巧妙融入当下的天气、时间和日程，起到心理疏导或氛围烘托的作用。

# Output Format (JSON)
请严格按以下格式输出，不要包含任何前后缀解释：
[
  {
    "id": 1,
    "song_name": "歌曲名",
    "artist": "歌手",
    "dj_script": "外面的雨下得有点大，周一的早晨总是让人有点紧绷。我知道你一会还有一个重要的交付会要开，别慌，先喝口咖啡。第一首歌，来自...的《...》，让它的旋律帮你稳住节奏。"
  },
  ...
]
```

## 🛠️ 进阶工程优化（让大脑更聪明）

为了让你的 AI 电台不流于表面，在开发大脑时可以加入以下设计：

### 1. 动态冷启动与“切歌”反馈

- **切歌（Negative Feedback）：** 如果用户在某首歌播了不到 10 秒就切歌了，把这条记录记入缓存。在下一次生成歌单时，告诉 LLM：*“用户刚刚切掉了《XX》，他现在可能不想听太吵/太悲伤的歌”*。

### 2. 日程的“提前量”感知

- 大语言模型具备很好的逻辑推理能力。你可以让它判断**日程的紧急程度和距离现在的时长**。
  - 如果日程是“1小时后有高压会议”，推荐**稳健、减压、专注**的歌。
  - 如果日程是“下班后去健身房”，在下班前 30 分钟的电台里，逐渐把曲风过渡到**节奏感强、多巴胺分泌**的歌。

### 3. TTS（文本转语音）的特殊标记

- 既然你已经接了 TTS 接口，为了让 AI DJ 听起来更自然，可以在 LLM 生成串词时，要求它加入一些**语气词**或者 **TTS 引擎支持的停顿标签**（如 `<break time="1s"/>`），甚至让它在串词的最后一句提示背景音乐淡入。
